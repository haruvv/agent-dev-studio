# AI自律開発エージェントチーム（MVP）要件定義書

本書は、Claude Code、GitHub、AWS 実行基盤、およびハーネスエンジニアリングの考え方に基づき、自然言語による要求受付から Pull Request 作成・検証・修正ループまでを段階的に自律実行可能な開発パイプラインの MVP 要件を定義するものである。

なお、本書において以下の用語を用いる。

- **本MVP**：本書で定義する自律開発パイプラインそのもの
- **開発対象**：本MVPが開発を進めるアプリケーション、サービス、または関連成果物
- **自律開発基盤**：本MVPを構成する受付機構、ワークフロー、運用ルール、設定、および実行基盤
- **agent-intake**：Discord から自然言語要求を受け付け、必要に応じて確認を行い、GitHub Issue を起票する受付コンポーネント
- **Orchestrator**：GitHub イベント、ジョブ状態、再試行、エスカレーションを制御するオーケストレーション機構
- **Worker**：実際の要件整理、設計、タスク分割、実装、修正等を実行するジョブ単位のエージェント実行環境

---

## 1. プロジェクト概要

### 1.1 目的

本MVPの目的は、AIエージェントが開発対象の開発を継続的かつ安定的に遂行できる実行環境を整備し、**「人間がチャットで自然言語の要求を送った場合に、一定条件のもとでレビュー可能な Pull Request まで自律的に到達できる状態」**を最小構成で実現することにある。

人間と自律開発パイプラインとの接点は、GitHub Issues の直接起票ではなく、Discord ボット（agent-intake）を介したチャットとする。人間は自然言語で要求を入力し、agent-intake がその内容を解釈して GitHub Issue を起票することで、以降の自律開発パイプラインが起動する。

本MVPでは、GitHub Actions に重い実装処理を集中させるのではなく、**GitHub を記録・状態遷移・レビューの中心としつつ、重処理は外部 Worker に委譲する構成**を採用する。

### 1.2 基本方針

本MVPは、以下の方針に基づいて構築する。

- **Issue を正史とすること**  
  Discord 上の自然言語要求は、そのまま実装に入るのではなく、一度 GitHub Issue に正規化して以降の処理の基準点とする。

- **GitHub を記録システムおよび協調基盤として運用すること**  
  要求、設計、実行計画、意思決定、Pull Request、レビュー結果、エスカレーション状態等を GitHub 上で追跡可能にする。

- **GitHub Actions はトリガーおよび CI を主用途とすること**  
  長時間実行、複数段階の自己修正、継続的な状態保持を要する重処理は外部 Worker に委譲する。

- **役割分担されたエージェントによる段階的処理を行うこと**  
  要件整理、設計、タスク分割、実装、修正などを段階的に進め、必要に応じて状態を引き継ぎながら前進させる。

- **人間は受付後の意思決定、最終判断、およびエスカレーション対応を担うこと**  
  人間は主として、自律開発基盤の整備、判断不能事項への対応、レビュー、統合判断を担う。

- **無制限な自動化を避けること**  
  自動再試行や自己修正には上限を設け、必要時には `blocked` または `needs-human` として停止可能であることを前提とする。

## 2. スコープ

### 2.1 本MVPで目指すこと

本MVPでは、最低限以下を成立させることを目標とする。

- Discord チャットを起点として、要求受付から GitHub Issue 起票までを自動化すること
- GitHub Issue を起点として、要件整理、設計、タスク分割、実装、Pull Request 作成までの段階的パイプラインを成立させること
- GitHub Actions をトリガーおよび CI に限定し、重処理を外部 Worker に委譲すること
- CI 失敗時に、一定回数まで自動修正ループを実施できること
- 停滞時または修正上限到達時に、人間へ適切にエスカレーションできること
- エージェントが参照可能な知識構造を GitHub リポジトリ内のドキュメントとして整理できること
- 最小限の検証と人間レビューにより、安全に前進可能であること

### 2.2 本MVPで対象外とすること

本MVPでは、以下は初期対象外とする。

- 初期段階からの完全無人運用
- あらゆる種類の開発タスクへの対応
- 高度な監視基盤、完全自動 UI 検証、完全自動本番反映
- 初期段階からの複雑な多段レビューエージェント運用
- 初期段階からの汎用的な全社横断エージェント管理基盤
- 無制限の自己修正ループ

## 3. 全体アーキテクチャ要件

### 3.1 レイヤ構成

本MVPは、以下の3層に分離して構成する。

| 層 | 役割 | 主な構成要素 |
|---|---|---|
| 受付層 | 人間からの自然言語要求受付、要求明確化、Issue 起票 | Discord, agent-intake |
| オーケストレーション層 | 状態管理、ジョブ発行、CI 結果処理、再試行制御、エスカレーション | GitHub Actions, Orchestrator, State DB |
| 実行層 | 要件整理、設計、タスク分割、実装、修正の実行 | ECS Fargate Worker |

### 3.2 GitHub の役割

GitHub は少なくとも以下を担うこと。

- Issue による正式な作業指示の保持
- Pull Request による成果物のレビュー単位の管理
- コメント、ラベル、チェック結果による進捗と状態遷移の可視化
- 関連ドキュメントおよび設計情報の保存
- 人間レビューおよび統合判断の基盤

### 3.3 GitHub Actions の役割

GitHub Actions は少なくとも以下に用途を限定すること。

- Issue または Pull Request 上のイベント検知
- 外部 Worker に対するジョブ発行
- CI 実行（lint、型検査、テスト等）
- 必要最小限の通知および外部連携起点

GitHub Actions 自体に長時間の実装処理、継続的ポーリング、複雑な自己修正ループを集中させないこと。

### 3.4 外部 Worker の役割

外部 Worker は少なくとも以下を担うこと。

- GitHub Issue または関連ドキュメントの読み込み
- 要件整理、設計、タスク分割、実装、修正の実行
- 必要なコミット、プッシュ、Pull Request 作成
- 実行結果の GitHub および State DB への反映
- 次工程へ進行するための状態遷移の実施

Worker はジョブ単位で起動・終了可能であることを前提とし、常駐前提を必須としない。

## 4. リポジトリ運用要件

### 4.1 マルチリポジトリ前提

本MVPでは、以下のようなマルチリポジトリ構成を許容する。

| リポジトリ種別 | 役割 |
|---|---|
| agent-intake リポジトリ | Discord 受付、Issue 起票、Webhook 処理等の基盤実装 |
| dev-agents リポジトリ | ワークフロー、ラベル定義、テンプレート管理 |
| 開発対象リポジトリ | 実際に AI エージェントが実装を行う対象 |

各リポジトリの責務は明確に分離し、役割の異なるコードや設定が曖昧に混在しないこと。

### 4.2 テンプレート展開

新規開発対象リポジトリを作成する場合、共通ワークフロー、必要ラベル、必要設定、Secrets 設定方針をテンプレートから適用可能であること。

### 4.3 GitHub 上の記録方針

GitHub 上には少なくとも以下を追跡可能な形で残すこと。

- 起票された要求
- 要件整理の結果
- 設計成果物
- タスク分割結果
- Pull Request
- CI 結果
- 修正履歴
- エスカレーション状態
- 重要な意思決定

## 5. ドキュメント要件

### 5.1 docs/ の位置づけ

`docs/` 配下のドキュメントは、説明資料ではなく、**エージェントが参照する記録システム**として扱うこと。

### 5.2 記録対象

`docs/` には、必要に応じて少なくとも以下を格納できること。

- 要件
- 設計
- 実行計画
- 意思決定ログ
- 運用ルール
- 制約一覧

### 5.3 要件と設計の記録方針

- 要件整理の初期段階では、Issue 本文およびコメントを対話の主な場として用いることを許容する
- 設計内容は、必要に応じてリポジトリ内の `docs/` 配下へファイルとして保存できること
- エージェントは Issue コメントのみではなく、`docs/` に記録された設計・計画・制約も参照可能であること

### 5.4 案内文書

エージェントが参照先を把握するための簡潔な案内文書を用意すること。  
当該案内文書は詳細仕様書ではなく、主要な参照先、構造、作業開始点を示す入口として維持すること。

## 6. 自律開発パイプライン要件

### 6.1 要求受付

- 人間は Discord のスラッシュコマンド等を通じて自然言語で要求を入力できること
- agent-intake は要求を解釈し、曖昧な場合は Discord 上で確認を求められること
- 明確化できた要求は GitHub Issue として起票できること
- 要求が曖昧なまま実装工程へ進まないこと

### 6.2 要件整理

- Issue 上の対話を通じて、要求内容の曖昧さを解消し、要件を確定できること
- 必要に応じて質問往復を許容すること
- 一定回数以上解消できない場合は、現時点の情報で確定するか、人間へエスカレーションできること

### 6.3 設計

- 確定した要件に基づき、設計書を作成できること
- 設計書には少なくとも、目的、実装方針、制約条件、作業分割方針、検証方法を含められること
- 設計成果物は、Issue コメントまたは `docs/` 配下のファイルとして参照可能であること

### 6.4 タスク分割

- 設計内容をもとに、1 Worker が 1 Pull Request で完結しやすい粒度まで作業を分割できること
- 分割結果は、子 Issue 等の形で GitHub 上に記録できること
- 親子関係が追跡可能であること

### 6.5 実装

- エージェントは作業単位に従い、実装、関連ドキュメント更新、必要な検証、および Pull Request 作成を実施できること
- 実装は専用ブランチ上で行うこと
- Pull Request には、少なくとも変更概要、関連 Issue、主要な判断結果が追跡可能な形で残ること

### 6.6 修正ループ

- CI 失敗時には、一定上限まで自動修正ループを実施できること
- 修正ループは、失敗内容を取得し、それに基づいてブランチ上で修正を継続できること
- 修正上限に到達した場合は、人間へエスカレーションできること

### 6.7 レビュー

- Pull Request は人間がレビュー可能な状態で提示されること
- MVPでは、人間レビューを統合前提条件として扱ってよい
- 将来的にレビュー補助エージェントを導入可能な構成であること

### 6.8 統合

- 所定の条件を満たした成果物のみを統合対象とすること
- 自動統合は、明示的な許可がある場合にのみ実施できること
- 明示的な許可がない場合は、人間判断待ちとすること

## 7. エージェントおよび処理責務要件

本MVPでは、少なくとも以下の役割を論理的に分離できること。

| 役割 | 主な責務 |
|---|---|
| 受付 | Discord 要求の受信、要求明確化、Issue 起票 |
| 要件整理 | Issue 上での要件整理および明確化 |
| 設計 | 設計書作成、制約整理、分割方針策定 |
| タスク分割 | 実装可能な粒度への分割 |
| 実装 | コード実装、関連ドキュメント更新、Pull Request 作成 |
| 修正 | CI 失敗時の修正ループ |
| オーケストレーション | ジョブ発行、状態更新、再試行、エスカレーション |
| 監視 | 停滞、失敗反復、上限超過の検知 |

これらは必ずしも別プロセス・別モデルである必要はないが、少なくとも責務として分離され、後から独立強化可能であること。

## 8. 状態管理要件

### 8.1 State DB

本MVPは、少なくとも以下の情報を保持できる State DB を持つこと。

- ジョブ識別子
- 対象リポジトリ
- Issue 番号
- ジョブ種別
- 現在状態
- Pull Request 番号
- ブランチ名
- 試行回数
- 修正回数
- 作成日時
- 更新日時

### 8.2 状態の用途

状態管理は少なくとも以下に利用できること。

- 二重起動防止
- 再試行回数管理
- 修正回数管理
- ジョブ進行状況の追跡
- Webhook イベントとの紐付け
- エスカレーション判断

### 8.3 停止状態

少なくとも以下の停止または保留状態を表現可能であること。

- `blocked`
- `needs-human`
- 人間回答待ち
- CI 修正上限到達
- 判定不能

## 9. ラベルおよびステート遷移要件

### 9.1 ラベルによる遷移

Issue および Pull Request の状態遷移を、少なくとも一部ラベルにより表現できること。

### 9.2 ラベル運用方針

- ラベルは、処理トリガー、進行中状態、停止状態、レビュー状態を表現可能であること
- ラベル設計は、読み手にとって意味が明確であること
- `blocked` および `needs-human` は、明示的な停止シグナルとして機能すること

### 9.3 無限ループ回避

ラベル更新によりイベントを連鎖させる場合、無限ループや重複起動を防ぐ設計であること。  
MVP では PAT 等を用いた制御を許容するが、将来的により安全な認証方式へ移行可能であること。

---

## 10. 実行基盤要件

### 10.1 受付基盤

- Discord インタラクションの即時応答制約に対応できること
- 要求処理本体は、必要に応じて非同期処理へ分離できること
- 曖昧要求に対する確認メッセージ送信が可能であること

### 10.2 ジョブ発行基盤

- GitHub Actions から外部 Worker に対してジョブ発行できること
- ジョブ発行は、キュー等の非同期手段を介して行えること
- ジョブ投入失敗時には検知可能であること

### 10.3 Worker 実行基盤

- Worker は、開発対象リポジトリを取得し、必要な CLI・認証情報を用いて実装を行えること
- Worker は、ジョブごとに使い捨て起動できること
- Worker 実行環境は、将来的に複数ジョブ種別へ拡張可能であること

### 10.4 イベント受信基盤

- CI 完了や Pull Request 状態変化を外部で受信可能であること
- 受信イベントに応じて、State DB 更新および必要な再修正ジョブ再投入が可能であること

## 11. 検証要件

### 11.1 決定論的ガードレール

- ツール実行後には、リンター、フォーマッター、型検査、テスト等の決定論的な検証を自動実行できること
- 検証結果は再利用可能な形で記録されること
- 失敗結果は、修正ループに利用可能であること

### 11.2 CI 段階的整備

本MVPでは、開発対象の成熟度に応じて段階的に以下を導入可能であること。

- ワークフロー定義および設定ファイルの妥当性確認
- リント
- 型検査
- 単体テスト
- 結合テスト
- ビルド
- データベース変更に関する検証

### 11.3 構造制約

- 開発対象における依存関係や配置ルールを、可能な限り機械的に検証可能であること
- MVPにおいては、最低限以下を対象とできること
  - レイヤ逆流の禁止
  - ドメイン外直接参照の禁止
  - 境界における型またはスキーマ検証の必須化
  - コード、設定、設計情報の配置ルールの明確化

---

## 12. 観測可能性要件

### 12.1 追跡可能性

人間が事後的に少なくとも以下を追跡可能であること。

- どの要求から始まった処理か
- どの Issue、どの Pull Request に対応するか
- どこで失敗したか
- 何回再試行したか
- どの時点で人間介入が必要になったか

### 12.2 取得対象

MVPでは、少なくとも以下のいずれかを機械的に取得可能であること。

- テスト結果
- ビルド結果
- CI 結果
- 実行ログ
- Pull Request 状態

### 12.3 将来拡張

将来的には、ログ、メトリクス、トレース、UI 操作結果等を自律的に確認可能な構成へ拡張できること。

---

## 13. 停滞監視および自動回復要件

- 開発または修正の進行が停滞した場合を検知可能であること
- 停滞判定は、進捗更新、成果物更新、応答状況、失敗反復等の複数シグナルを総合的に評価できること
- 停滞時には、再試行、処理再開、必要に応じた人間へのエスカレーションを実施可能であること
- 自動回復には上限を設け、無制限な再試行を防止すること
- 必要に応じて、ドキュメント、制約、検証手段等の不足を補うための改善タスクを起票可能であること

---

## 14. 統合条件

本MVPが生成した Pull Request の統合は、以下の条件をすべて満たした場合にのみ実施する。

- **レビュー可能状態であること**  
  Pull Request とその関連情報が人間にとってレビュー可能であること。

- **検証通過**  
  定義済みの CI、ビルド、テスト、リント等が成功していること。

- **停止状態でないこと**  
  `blocked` その他の停止状態に該当していないこと。

- **明示的な統合許可**  
  自動統合を許可する明示的な条件または指示が存在すること。  
  当該条件または指示が存在しない場合は、人間による確認待ちとする。

---

## 15. 非機能要件

### 15.1 再現性

同一入力に対し、可能な限り同一の検証結果および同一の実行手順が得られること。

### 15.2 可観測性

本MVPにおけるエージェントの失敗理由を、人間が事後的に追跡可能であること。

### 15.3 分割可能性

大きな要求を、安全に作業単位または複数の Pull Request に分割可能であること。

### 15.4 エスカレーション可能性

自律処理が困難な場合に、曖昧な状態で停止するのではなく、人間へ適切に引き渡し可能であること。

### 15.5 拡張可能性

MVP の構成を保ったまま、将来的に以下へ拡張可能であること。

- レビュー補助エージェントの追加
- 統合判定自動化の強化
- マルチリポジトリ展開の強化
- Durable Execution 導入
- 観測可能性の高度化
- より安全な認証方式への移行

---

## 16. 運用方針

### 16.1 修正優先の運用

開発の進行においては、長時間の停止よりも、小さな修正を繰り返しながら前進することを優先する。  
ただし、無制限な試行は許容せず、再試行上限およびエスカレーション条件を定義すること。

### 16.2 記録優先の運用

重要な判断、設計変更、制約、停止理由は、後から追跡できる形で GitHub または `docs/` に記録すること。

### 16.3 ガベージコレクション

- 不要コード、重複ドキュメント、陳腐化したルール、技術的負債候補を定期的に検出可能であること
- 検出結果に基づき、改善またはリファクタリングのためのタスクを起票可能であること
- ドキュメントの鮮度および参照整合性についても定期点検の対象とすること

---

## 17. MVPの到達条件

本MVPは、少なくとも以下が実現された時点で成立したとみなす。

- Discord 上の自然言語要求から GitHub Issue が起票されること
- GitHub Issue を起点として、要件整理、設計、タスク分割、実装、Pull Request 作成までが段階的に進行すること
- GitHub Actions はトリガーおよび CI として機能し、実装本体は外部 Worker が担うこと
- CI 失敗時に一定回数まで自動修正が行われること
- 上限超過時に `needs-human` 等で停止し、人間へ引き渡せること
- 人間が最終的に Pull Request をレビューし、統合可否を判断できること
